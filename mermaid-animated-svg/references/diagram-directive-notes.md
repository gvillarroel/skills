# Mermaid Diagram Directive Notes

Use this reference after `animation-directives.md` when choosing selectors, points, marks, and timing for a specific Mermaid diagram type. These notes come from the local Mermaid fixtures in `examples/mermaid/`, the generated SVG gallery in `examples/mermaid-svg-animated/`, and diagram-focused agent reviews.

The repository also includes a generated by-type directive collection in `examples/mermaid-animation-directives/by-type/`. Each source file copies one canonical Mermaid example and appends `%% @animate` comments. `CRITIQUE.md` in that directory records the selector tradeoff or language gap exposed by each example.

## General Rule

Keep the language small. Mermaid owns the diagram. Directives should select rendered SVG elements, define reusable SVG-space points, add overlay marks, and orchestrate time. Prefer stable `id:`, `#id`, `text:"..."`, `.class`, `role:node`, `role:edge`, and narrow `css:"..."` selectors from `--list-elements`.

Semantic selectors such as `state:Review`, `edge:A->B`, `plot("X", 10)`, or `message:"Submit"` are useful future syntax, not current syntax.

After generating an example, critique it against rendered output:

- Does every selector resolve to the intended count?
- Does every point expression resolve to exactly one element?
- Does the final frame preserve Mermaid geometry?
- Did a short `text:"..."` selector overmatch?
- Did a class selector match multiple shapes for one concept?
- Did a relation ID direction differ from the source notation?

If the answer exposes a recurring issue, update either the example, this reference, or the core directive reference.

## Diagram Notes

| Diagram type | Strong current selectors | Best animation pattern | Current gaps or future syntax |
| --- | --- | --- | --- |
| Architecture | Service and edge IDs such as `id:my-svg-service-renderer` and `id:my-svg-L_renderer_browser_0`. | Reveal groups and services first, trace exact edges, move artifact marks between service anchors. | Future `service:` and `edge:source->target`; current fixture has overlapping source/markdown positions, so grouped animation reads better. |
| Block | Alias-derived IDs such as `id:my-svg-source`, `id:my-svg-arrow1`, and connection IDs like `id:my-svg-1-source-markdown`. | Reveal the pipeline in sequence, move a payload mark across blocks, then color or pulse result blocks. | Future `block:` and `block-edge:` selectors; avoid `role:edge` for one connector. |
| C4 | `text:"Skill Author"`, `text:"Skills Repository"`, `text:"Mermaid Renderer"`, and relationship label text. | Reveal boxes, reveal the coarse relationship group, then move a packet mark through box anchors. | Individual C4 relationships are not stable targets yet; future `person:`, `container:`, and `relationship:` selectors. |
| Class | Exact class IDs from `--list-elements`; `.class-diagram-class`, `.class-diagram-relation-line`, and relation label text for broad groups. | Reveal classes first, trace relations after both endpoints are visible, then move dependency marks. | Future `class:Product` and `relation:Product->PricePoint`; text selectors can overmatch names like `Product` and `ProductStatus`. |
| Entity Relationship | Entity IDs such as `id:my-svg-entity-CUSTOMER-0`, relationship IDs, `role:edge`, and unique relationship label text. | Reveal entity, trace relationship, reveal label, then pulse or color the target entity. | Future `entity:` and `relationship:` selectors; attribute-row targeting is weak. |
| Event Modeling | `text:"..."` for boxes; `.em-box` and `.em-relation` only for broad groups. | Keep swimlanes static, reveal boxes in causal order, move a token between box anchors, pulse produced events. | Future event-modeling edge selectors; current arrows lack semantic source/target hooks. |
| Flowchart | Unique `text:"..."` node labels, `id:` from listed elements, classes for groups, and `role:edge` for all edges. | Reveal nodes, trace edges after endpoints, move request marks along node anchors and path midpoints. | Future `edge:A->B` and nested groups; subgraph cluster IDs can be unstable. |
| Gantt | Task IDs such as `id:my-svg-src`, labels via `text:"Draft sources"`, section classes, exclusion IDs, and `line.today`. | Reveal section rows/title first, then task bar plus label from left to right; use progress marks without moving bars. | Future `task:`, `section:`, and `date:` selectors; row backgrounds are broad class-only rectangles. |
| GitGraph | Commit classes such as `css:.init`, `css:.mmd`, `css:.commit-merge`, branch labels, and branch arrow classes. | Reveal commits left to right, move a release token between commit anchors, then color or pulse merge/verify commits. | Future `commit:`, `branch:`, and individual edge selectors; avoid generated hash classes. |
| Ishikawa | Cause and problem labels with `text:"..."`; `.ishikawa-spine`, `.ishikawa-branch`, and `role:edge` for broad structure. | Reveal problem/head and spine first, then one cause branch at a time, using marks for root-cause callouts. | Future `cause:` and `branch:` selectors; per-branch targeting is awkward because branches share classes. |
| Journey | Exact task and section text; `.journey-section-column` and `.journey-task-column` for broad columns. | Reveal each section header with its first task, then task columns left to right; move a progress dot between tasks. | Future `section:` and `task:` selectors; no stable IDs are emitted in the fixture. |
| Kanban | Stable IDs such as `#my-svg-backlog`, `#my-svg-progress`, `#my-svg-colors`, and `#my-svg-validator`. | Reveal columns together, reveal cards in sequence, move a ring/dot mark across columns. | Future `column:` and `card:` selectors; do not move real cards unless changing the final board layout is intended. |
| Mindmap | Node text for readable targets and explicit edge IDs such as `#my-svg-edge_0_4`. | Reveal root, trace a branch edge, reveal branch node, then reveal children in sequence. | Future `mindmap-branch:` and `edge:root->child`; broad `role:edge` selects every connector. |
| Packet | Full rendered row text and absolute `xy(...)` field centers. | Reveal rows, then move a scanner ring through field centers with short callouts. | Future `field:`, `byte:`, and `row:` selectors; duplicate short labels make `text:"Content hash"` ambiguous. |
| Pie | Segment classes such as `.pie-label-validation`, `.pie-value-24`, and `.pie-segment-0`; chart center from `.pieOuterCircle`. | Reveal complete segments, then add callout labels near the chart or pulse a segment group. | No selector intersection; category classes often resolve wedge, percent label, and legend together. |
| Quadrant | `text:"Point label"` for points; `.quadrant-chart-point` and `.quadrant-1` through `.quadrant-4` for broad groups. | Reveal points in sequence, move a ring/dot decision mark between point centers, color the selected point. | Future `point:"Label"`, `plot(x,y)`, `.symbol`, `.label`, and `.group-center`; quadrants stay static. |
| Radar | Rendered curve classes like `css:".radarCurve-0"`, axis/legend text, and base classes such as `.radarGraticule`. | Reveal frame first, reveal series in z-layer order, then pulse or label a series near its legend. | Future `radar-series:`, `radar-axis:`, and vertex selection; per-vertex points are not exposed. |
| Requirement | Exact requirement and relation IDs, such as `id:my-svg-generalized_animation` and relation IDs from `--list-elements`. | Reveal requirements and elements, trace links after endpoints, move verification marks, then show status labels. | Future `requirement:`, `element:`, and `relation:` selectors. |
| Sankey | Node IDs or label text for nodes; `role:edge` for all links. | Reveal source/target columns, trace flows, move packet marks between node anchors, pulse destinations. | Future `link:"Source->Target"`; individual link targeting is weak today. |
| Sequence | Participant IDs from `--list-elements`, `role:edge` for messages, and participant anchors with offsets. | Reveal participants, trace messages, move request/response marks between participant anchors. | Future `participant:`, `message:1`, `message:"Submit"`, and synchronization helpers such as `all(a.end,b.end)`. |
| State | Unique state text or exact state IDs; exact IDs for start/end markers. | Move a token through state anchors, use label marks for dwell, and color the active state. | Future `state:`, `transition:`, `wait`, and `dwell`; pulse must preserve Mermaid group transforms. |
| Timeline | `text:"..."` for sections, dates, and events; `.timeline-event`, `.timeline-date`, and `.timeline-line` for groups. | Draw lines first, reveal sections/dates left to right, reveal event cards in sequence, move a progress mark. | Future `section:`, `date:`, and `event:` selectors; avoid reusable `#my-svg-node-*` IDs. |
| Treemap | Section/leaf text and simple leaf/section classes as fallback. | Reveal hierarchy sections first, then leaves, then highlight the final leaf with a ring or label. | Future `treemap-section:` and `treemap-leaf:` selectors; tiny hidden leaves may select but not show useful animation. |
| Treeview | Exact label text; `.treeView-node-label` for all labels and `css:"line.treeView-node-line"` for all connectors. | Reveal root/folders, reveal files, then scan selected leaves with a ring mark. | Future `tree-node:` and `tree-edge:` selectors; per-line connectors are not stable. |
| Venn | Generated `.venn-sets-*` classes from `--list-elements`; readable text labels as fallback. | Reveal base sets, pairwise intersections, then triple intersection; use a ring/label on the final overlap. | Future `venn:set` and `venn:intersection`; attribute selectors are not available through `css:`. |
| XY Chart | `.chart-title`, `.bottom-axis`, `.left-axis`, `.bar-plot-0`, `.line-plot-1`, `.line-plot-2`, and category text. | Reveal title/axes first, bars next, line series after bars; use marks for chart-level callouts. | Future `plot("category", value)` and `series:"name"`; numeric text labels collide with ticks. |
| Wardley | Use visible component labels and broad map classes from `--list-elements`. | Reveal the map frame and components by value-chain flow, then move a focus mark along evolution positions. | Future `component:`, `evolution:`, and map-space points. |
| ZenUML | Prefer participants and message text from `--list-elements`; use sequence-style anchors. | Reveal participants first, trace call groups, then move request marks down the interaction stack. | Future ZenUML-specific `participant:` and `message:` selectors; rendered output can be coarse. |

## Current Language Core

```text
%% @animate target <name> = <selector>
%% @animate group <name> = <target>, <target>
%% @animate point <name> = <target>.<anchor> | xy(x,y) | mid(a,b) | offset(p,dx,dy)
%% @animate mark <name> at <point> shape=dot|ring|label ...
%% @animate at <time> reveal|trace|move|color|pulse|hide|set <subject> ...
```

Use `.end` references and named actions to orchestrate several diagrams at once. Prefer overlay marks for movement when the Mermaid geometry should remain stable.
